# План работ — переписывание сайта (data-driven rewrite)

Цель
- Переписать сайт на новую архитектуру: data-driven статический сайт с общими шаблонами и дизайн‑токенами.

Хостинг и технологии
- Хостинг: GitHub Pages (фиксирован).
- Технологии: статический HTML/CSS/JS, данные в JSON/JS модулях (demo-data). Деплой через GitHub Actions.

Коротко о подходе
- Архитектура: шаблоны + data-driven рендеринг (sites/newdesign или pages/rewrite).
- CSS: единый набор токенов в css/main.css; минимальные страницы без сборщика, чтобы проще деплоить на Pages.
- Процесс: итеративные feature‑ветки → PR → merge в main → автодеплой.

Область первой волны (минимум релиза)
- Главная (index)
- TI demo (dashboard, schedule)
- Далее по очереди: career, benefits, events, assistant, grades

Роли
- Ведущий: вы — принимаете решение по приоритетам и дизайну.
- Исполнитель: я — подготовлю патчи, ветки и PR по вашему подтверждению.
- CI/Релиз: GitHub Actions → GitHub Pages.

Стратегия ветвления
- Feature ветки: `rewrite/<page>` или `rewrite/<module>`.
- Каждый значимый набор изменений — отдельный PR (не монолитные изменения).
- PR проходит CI и ревью, затем merge в `main`. По согласованию возможен прямой коммит в `main` для критических исправлений.

Пошаговая последовательность работ
1) Подготовка (сейчас выполнено часть)
   - Архивировать текущее состояние: `tools/archive_current_pages.sh` → `docs/archive/legacy-<timestamp>.tar.gz` (архив создан).
   - Убедиться, что все статические ассеты на месте (images/, favicon).
   - Проверить, что `main` синхронизирован с origin: `git checkout main && git pull origin main`.

2) Скелет новой архитектуры (уже в репозитории)
   - Проверить и зафиксировать шаблон: `sites/newdesign` (index.html, site-content.js, site-render.js).
   - Вынести общие CSS‑токены в `css/main.css`, удалить критические inline‑стили.

3) Миграция страниц (итеративно, одна страница = одна ветка/PR)
   - Для каждой страницы:
     a) Создать ветку: `git checkout -b rewrite/<page>`.
     b) Описать data model: `demo-data/<page>.js`.
     c) Реализовать страницу через шаблон (sites/newdesign или pages/rewrite).
     d) Локально проверить, открыть в браузере или через простой статический сервер.
     e) Коммит → push → открыть PR.

4) Ревью, CI и деплой
   - PR проходит CI (Actions) и ревью. Исправляем замечания.
   - После merge в `main` GitHub Pages деплоит новую версию.

PR‑чеклист
- Описание цели PR.
- Скриншоты/preview (если UI меняется).
- Добавленный/обновлённый data model (demo-data).
- Проверка ассетов (нет 404).
- Минимальная accessibility‑проверка (alt, семантика).

Критерии приёмки страницы
- Корректное отображение на мобильных и desktop.
- Отсутствие 404 на ассеты.
- CI зелёный и Pages успешно деплоит сайт.

Короткий план на 2 недели (итерации)
1. День 1: Архив и утверждение плана (этот документ).
2. День 2–3: Подготовить дизайн‑токены и базовые компоненты (docs/design-system).
3. День 4–6: Миграция главной страницы (ветка rewrite/home → PR → merge).
4. День 7–10: Миграция TI demo (dashboard + schedule).
5. День 11–14: Миграция второй волны (career, benefits, events).
